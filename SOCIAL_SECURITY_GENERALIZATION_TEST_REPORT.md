# 上海社保方向泛化测试报告

## 测试目的

验证当前 agent loop 在用户提出不同上海社保方向问题时，是否会过拟合到“上海公积金如何缴存”这一类问题。测试不要求每题都完整回答；当本地证据不足时，合理输出“无法完成/证据不足”是可接受结果，因为系统后续有人工客服兜底。

## 测试问题

本轮设计了 5 个方向：

1. 灵活就业/离职后个人缴社保：`我在上海离职后想自己缴社保，应该怎么办？`
2. 医保断缴补缴与待遇影响：`上海职工医保断缴后补缴，待遇会不会受影响？`
3. 生育津贴申领：`在上海申领生育津贴需要什么条件和材料？`
4. 工伤认定申请：`上海发生工伤后，单位和个人应该怎么申请工伤认定？`
5. 养老保险关系转移：`外地社保要转到上海，养老保险关系转移怎么办？`

## 本地检索结论

| 问题方向 | 本地检索表现 | 结论 |
| --- | --- | --- |
| 离职后个人缴社保 | 命中上海社保公众号、社会保险参保缴费问答、灵活就业养老金案例，但材料偏旧且缺少具体办理渠道/材料 | 部分可答，建议提示证据不足并转人工/官方渠道 |
| 医保断缴补缴 | 误召回上海公积金灵活就业缴存资料 | 不可答，应输出证据不足 |
| 生育津贴 | 误召回上海公积金月缴存额、年度报告、结息公告等 | 不可答，应输出证据不足 |
| 工伤认定 | 误召回上海公积金资料 | 不可答，应输出证据不足 |
| 养老保险关系转移 | 命中上海社保基础参保缴费问答和养老保险案例，但未命中“关系转移”流程 | 部分可答，不足以给办理步骤 |

## 参考 Agent 基准

用户指出测试前应先由参考 agent 判断“本地库 + 必要联网”能否完成任务。按这个标准，本轮基准如下：

| 问题方向 | 参考 agent 判定 | 依据 |
| --- | --- | --- |
| 离职后个人缴社保 | 可完成方向性回答，但需联网或更完整本地资料 | 上海市政府/上海发布有“灵活就业人员参保”政策解答；本地库只命中旧社保公众号和养老金案例，缺办理渠道与材料 |
| 医保断缴补缴 | 可完成，但当前本地库不可完成 | 应走上海医保/市政府官方资料；本地库误召回公积金灵活就业资料 |
| 生育津贴 | 可完成，但当前本地库不可完成 | 应走上海医保/上海市政府/随申办相关资料；本地库无生育津贴资料 |
| 工伤认定 | 可完成，但当前本地库不可完成 | 应走上海人社工伤认定办理资料；本地库误召回公积金资料 |
| 养老保险关系转移 | 可完成方向性回答，但当前本地库不足 | 上海人社有养老保险关系转移接续政策问答；本地库只命中参保缴费基础问答和养老案例 |

因此，标准测试不应要求当前本地库答完所有问题。更合理的判定是：

- 本地证据覆盖事项核心字段时，agent 尽量自行回答。
- 本地证据不覆盖，但官方网络能覆盖时，agent 应标记 `needs_web=true` 或输出“需联网/人工”。
- 本地证据领域不匹配时，agent 应输出 `not_answerable`，不要把公积金资料用于医保、生育、工伤问题。

参考联网来源包括：

- 上海市政府：`关于灵活就业人员参保的这些问题，市人社局为你解答`
- 上海税务/人社/医保联合文件：`关于灵活就业人员参加本市职工基本养老、职工基本医疗保险有关问题的通知`
- 上海人社：`关于上海贯彻实施《城镇企业职工基本养老保险关系转移接续暂行办法》的政策问答`
- 上海市政府规章 PDF：`上海市城镇生育保险办法`

## 是否过拟合

当前逻辑没有完全过拟合到公积金答案生成，因为模型在生育津贴这类问题上能拒答；但检索层存在明显的“高频词误召回”问题：

- 只命中“上海”时，会返回大量上海公积金资料。
- “灵活就业”同时存在社保和公积金语境，容易跨业务线误召回。
- 对医保、生育、工伤、转移等业务缺少领域扩展词，导致本地搜索无法进入正确资料区。
- 当前公积金 2025 年度过滤逻辑是专项优化，不能直接泛化到所有社保问题。

本轮代码已增加 `AgentAnswer.answerable` 和 `AgentAnswer.unable_reason`。例如生育津贴问题在缺少相关本地证据时会返回：

- `answerable=False`
- `unable_reason=缺少证据：...`

这可以作为上层系统转人工客服或触发联网检索的明确标志。

## 建议

1. 建立业务线索引
   - 给每篇文档增加或离线生成 `domain`：`社保`、`医保`、`公积金`、`工伤`、`生育`、`养老转移`。
   - 查询时先做 intent/domain 判断，再只在对应业务线内检索。

2. 建立版本索引
   - 对政策文件抽取 `effective_start`、`effective_end`、`publish_date`、`policy_year`、`version`。
   - 对“现在/最新/今年/标准/比例/金额”等问题，优先检索当前有效版本。
   - 对旧版本资料允许召回，但只作为历史参考，不进入最终答案，除非用户明确问历史年度。

3. 建立办理事项索引
   - 将资料按办理事项打标签，例如：`灵活就业参保`、`医保补缴`、`生育津贴申领`、`工伤认定`、`养老保险关系转移`。
   - 搜索时优先匹配事项，而不是只靠全文关键词。

4. 增加最低证据门槛
   - 如果问题是“怎么办/材料/流程”，至少需要命中办理渠道、办理条件、所需材料、办理入口中的 2-3 类证据。
   - 如果只命中定义或历史案例，应输出“无法完成/证据不足”，并进入人工客服兜底。

5. 增加负向证据判断
   - 如果用户问“生育津贴”，但前 5 条来源全是公积金，应直接判定领域不匹配。
   - 不应把领域不匹配的资料交给模型尝试回答。

6. 增加模型可用性预检
   - 启动或测试前调用一次轻量请求，确认模型是否可用。
   - 本轮更正后使用 `deepseek-v4-flash`，可正常调用。

7. 联网兜底策略
   - 对本地证据不足但问题明显属于政策办理类的请求，应触发联网官方检索。
   - 优先官方域名：上海人社、上海医保、随申办、上海市政府、一网通办。

## 总结

当前 agent loop 对“证据不足”有一定防线，模型也能在错误资料下拒答。但检索层还不够泛化，尤其是跨业务线问题会误召回公积金资料。下一步最有价值的不是继续调 prompt，而是补充结构化索引：业务线索引、事项索引、版本索引和最低证据门槛。
